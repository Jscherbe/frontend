# Idea: `@ulu/frontend` MCP Server

## Overview

Instead of relying on AI agents to scrape documentation web pages (which is token-heavy, prone to parsing errors, and inefficient), we should build a dedicated **Model Context Protocol (MCP)** server for the `@ulu/frontend` library. 

This server would act as a bridge between a developer's local AI (e.g., Cursor, Claude Desktop, Gemini CLI) and the library's internal structures, providing direct, machine-readable access to components, APIs, and styling conventions.

This could easily be started by users via `npx @ulu/mcp-server` or configured in their AI tools.

## Why MCP?

1.  **Token Efficiency & Accuracy:** The AI receives exact HTML/SCSS/JS snippets and API signatures without the "noise" of documentation site wrappers (navs, footers, etc.).
2.  **Tool-Calling:** Agents don't have to guess where to look; they are provided explicit tools (e.g., `get_component_markup("accordion")`).
3.  **Automation & Introspection:** Much like how the current API docs are generated, the MCP server could dynamically parse the source code (or the generated docs JSON) to serve up-to-date information.
4.  **Enforces Rules:** It provides a direct channel to enforce the strict BEM conventions, the prohibition of Tailwind/Bootstrap, and the required use of `data-ulu-*` attributes.

## Potential Capabilities (Resources & Tools)

### Resources (Static Context)
These act as internal URIs the agent can read to understand the framework's philosophy before writing code.
*   `ulu://architecture/rules`: Serves the core principles (no Tailwind, strict BEM, separate JS/CSS).
*   `ulu://conventions/bem`: Explains the `__` and `--` structure and explicitly forbids SCSS concatenation (e.g., `&__element`).

### Tools (Functions the Agent Can Call)
These allow the agent to query exactly what it needs, when it needs it.

*   `get_component_markup(componentName)`
    *   **Purpose:** Returns the exact, framework-compliant HTML structure for a component (e.g., "tabs", "modal").
    *   **Details:** Ensures the agent includes the correct `data-ulu-component` attributes and base classes without guessing.
*   `get_component_js_api(componentName)`
    *   **Purpose:** Returns the initialization signature, available methods, and configuration options for a component's JavaScript behavior.
*   `search_utility_classes(category)`
    *   **Purpose:** Allows an agent to query native utility classes (e.g., "spacing", "typography") to find classes like `margin-top-large` or `type-small`.
    *   **Details:** Crucial for preventing the agent from hallucinating Tailwind classes.
*   `get_css_variables(moduleName)`
    *   **Purpose:** Lists the configurable SCSS variables for a specific module, allowing the agent to theme components safely using the existing system.

## Implementation Strategy (Automated Generation)

To avoid maintaining duplicate information, the MCP server should be powered by the existing automated systems that generate the site's API documentation (Eleventy, JSDoc, SassDoc).

1.  **Hook into Existing Plugins:** We already use `@ulu/sassdoc-to-markdown` and custom Eleventy plugins (`jsdocPlugin`, `sassdocPlugin`, `pluginCodePreview`). We can tap into the raw JSON/AST data generated by these tools *before* it is converted to HTML/Markdown for the website.
2.  **Extract SCSS & JS Data:**
    *   **SCSS:** Programmatically run SassDoc (or hook into its data pipeline) to extract variables, default values, and descriptions (e.g., component configuration properties).
    *   **JS API:** Programmatically run JSDoc to extract initialization signatures, parameters, and class methods for UI components.
3.  **Extract HTML Markup:** We can use Eleventy's data cascade or write a custom Node script (using something like `cheerio`) to parse the demo pages (`site/pages/demos/`). We can locate the actual component HTML by searching for the `data-ulu-component` attributes and extract only that relevant boilerplate, stripping away the documentation site wrappers.
4.  **Generate Artifact:** Create a build script (e.g., `npm run build:mcp-artifact`) that merges all this extracted data into a single, highly structured JSON file (e.g., `dist/mcp-api.json`).
5.  **Serve Artifact:** The actual `@ulu/mcp-server` package will be extremely lightweight. It won't parse any files; it will simply load the generated `mcp-api.json` and expose it via the standard MCP Tool endpoints (like `get_component_markup()`). Every time the documentation is built, the MCP server's knowledge base will automatically update.
